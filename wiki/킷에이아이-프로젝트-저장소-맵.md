### 4. newsletter

**한 줄 요약:** 웹 중심 AI/자동화 뉴스레터 사이트 — 이메일은 프리뷰, 전체 읽기는 웹으로 유도

| 영역 | 기능 |
|---|---|

**DB 스키마 (Supabase Postgres, 7개 테이블):**
`profiles` (관리자) · `categories` · `issues` (호, Tiptap JSON) · `issue_categories` · `media` · `subscribers` · `send_logs`

**아키텍처:**
```
[브라우저] → [Vercel: Next.js App]
               ├── public routes (/, /issues/[slug], /categories/[slug], /archive)
               └── admin routes (/admin/*)
                     ↓
              [Supabase]       [Resend]
               ├─ Postgres      └─ 트랜잭션 이메일
               ├─ Auth (매직링크)
               └─ Storage (이미지)
```

**기술 스택:** Next.js App Router, TypeScript, Tailwind CSS, Supabase, Resend, Tiptap 에디터

**GitHub 설명:** `AI/automation newsletter — web-first publication`
**최종 업데이트:** 2026-05-14
**상태:** MVP 구현 중 (Next.js 스캐폴딩 + Supabase 마이그레이션 완료)

> **[newsletter-agent](#12-newsletter-agent)** 와 짝을 이룸 — 에이전트가 콘텐츠를 생성하면 이 사이트의 Supabase에 직접 INSERT. 두 저장소가 하나의 뉴스레터 시스템을 구성한다.

```
2. select_topic Claude: 아이템 + 기발행 호 제목 → 테마·앵글·소스 3~6개 선정 (중복 회피)
        │
3. outline      Claude: 테마 + 소스 요약 → 섹션 개요 JSON
        │
4. draft        Claude: 개요 + 소스 본문 → 한국어 마크다운 에세이 (1,500~2,500자)
        │
5. metadata     Claude: 본문 → title, subtitle, slug, excerpt, categorySlugs
        │
6. review       터미널 미리보기 → 사용자 승인 / 재시도 / 다른 주제 / 취소
        │
7. publish      Supabase INSERT → newsletter 사이트 issues + issue_categories 테이블
```

**RSS 소스 (4개):**
- `anthropic.com/news/rss.xml`
- `openai.com/blog/rss.xml`
- `blog.google/technology/ai/rss/`
- `techcrunch.com/category/artificial-intelligence/feed/`

**핵심 설계 원칙:**
- **승인 게이트 코드 강제**: `publish`는 `review`가 `{approved: true}`를 반환해야만 호출. LLM이 자동 게시하는 경로 없음
- **중간 결과 영속화**: `runs/<ISO-timestamp>/<step>.json` — 단계별 재개 가능
- **Zod 스키마 검증**: 모든 LLM 출력 타입 검증 + 실패 시 최대 3회 재시도
- **함수형 파이프라인**: 각 단계 `(input) => Promise<output>`, 전역 상태 없음
- **LlmClient 추상화** (2026-05-18 추가): `src/lib/llm.ts` 에 공급사 중립 인터페이스 → Claude↔Gemini 교체 1파일

**LLM 마이그레이션 이력:**
- v1: Claude API (Anthropic SDK, 프롬프트 캐싱)
- v2 (2026-05-18): Gemini API로 전환 (`GEMINI_API_KEY` → 기본값), `LlmClient` 인터페이스 추출로 공급사 독립화
- `lib/anthropic.ts`는 유지 — `ANTHROPIC_API_KEY` 설정 시 다시 Claude로 전환 가능

**추가 기능 (2026-05-18):**
- `covers.ts` 신규: 발행 후 커버 이미지 자동 생성 (cli.ts에서 publish 후 자동 호출)

**newsletter와의 연동:**
- `lib/tiptap.ts`: 마크다운 → Tiptap doc JSON 변환 (newsletter 에디터 포맷)
- `lib/supabase.ts`: service_role 키로 newsletter DB 직접 접근
- `AGENT_AUTHOR_ID` 환경 변수로 게시 작성자 지정

**기술 스택:** TypeScript, Node.js (tsx), Gemini API (`@google/genai`), Supabase, Zod, rss-parser, Tiptap

## 비즈니스 관점 인사이트

### 인사이트 1: 자동화 에이전트 실증 포트폴리오가 이미 완성되어 있다

AI바우처 신청, AX 에이전시 영업 시 가장 강력한 무기는 "이미 만들어서 쓰고 있는 것들":
- `ai-agent` — 월 60시간 절감 (6개 클라이언트 운영 중)
- `ecount_kitt` — 8개 카테고리 37개 워크플로우 (제안서 완료)
- `invoice` — 실제 견적서 자동화 도구

### 인사이트 2: Docufinder는 별도 수익화 가능한 완성 제품

v2.5.7 출시, Windows/Mac 지원, BSL 라이선스. 기업용 폐쇄망 문서 검색으로 포지셔닝 가능. AI바우처 수요기업에 번들 제안하거나, 독자적 SaaS 판매 트랙 구성 여지 있음.

### 인사이트 3: meta_report + ai-agent = 광고 AX 에이전트 패키지

두 저장소를 하나의 "메타광고 AX 패키지"로 묶으면 판매 가능한 솔루션:
- ai-agent: 자동 리포트 생성 (노션 업로드)
- meta_report: 클라이언트용 대시보드 UI
→ kitt PolicyDoc Agent와 동일한 구조: **검증된 파이프라인 + 새 UI**

### 인사이트 4: newsletter는 PRD가 아닌 실제 코드가 MVP 진행 중

이전에는 PRD 단계로 알고 있었으나, 실제로 Next.js 스캐폴딩 + Supabase 마이그레이션까지 완료된 상태. 설계 문서(2026-05-11/14)도 완성도가 높아 조기 론칭 가능. 마케팅(AX 에이전시 플랜) → newsletter 발행 → 구독자 → AX 문의 전환 퍼널이 이미 준비 중.

### 인사이트 5: newsletter + newsletter-agent = AX 마케팅 플랜의 핵심이 이미 코드로 존재

[[kitt-ax에이전시-마케팅-플랜]]에서 "뉴스레터 채널 구축"을 목표로 잡았는데, 실제로 두 저장소가 이미 완성 단계다:
- `newsletter`: 웹 퍼블리케이션 + 구독 + 관리자 UI
- `newsletter-agent`: RSS → Gemini/Claude 자동 초안 → 승인 → 게시 7단계 파이프라인 (LlmClient 추상화로 공급사 교체 가능)

남은 것은 발행 시작뿐. "계획"이 아닌 "작동하는 시스템"이 이미 있다.

### 인사이트 6: parking은 예상 밖의 완전한 독립 제품

"주차 관련 서비스일 것"이라고 예상했지만 실제로는 **인천공항 전용 발렛 파킹 플랫폼**. 고객 앱(RN) + 기사 앱(RN + Android Kotlin) + 관리자 웹(Next.js) 3앱 모노레포 구조로 상당한 개발 규모. 킷에이아이의 기술 범위가 AI 자동화를 넘어 모바일 앱까지 포괄함을 보여준다.

### 인사이트 7: newsletter-agent는 AI바우처/AI챌린지 기술 실증의 직접 근거

[[AI오픈데이터챌린지-사업계획서-초안]]에서 "LLM + RAG + 자동 생성 파이프라인"을 강조하는데, newsletter-agent가 그 실증이다. RSS 수집 → Gemini/Claude 초안 → Zod 검증 → DB 게시 전 파이프라인이 실운영 코드로 존재. "우리는 이미 만들었습니다"의 증거로 사용 가능. 특히 **LlmClient 추상화**는 "Claude/Gemini/GPT 등 LLM 공급사 선택 가능한 파이프라인"이라는 기술 차별점으로 활용 가능.

### 11. parking

**한 줄 요약:** 인천공항(ICN) 전용 스마트 발렛 파킹 플랫폼 — 사용자·기사·관리자 3앱 모노레포

**서비스 시나리오:**
```
고객 (user-app)         기사 (driver-app)          관리자 (admin-web)
    │                         │                          │
앱으로 발렛 예약          배정 알림 수신              예약 현황 실시간 모니터
차량 픽업 장소 선택 →    지정 픽업 장소 이동 →       주차 구역/면 배정
귀국 후 차량 반환 요청   주차 완료 처리             기사 관리·통계
```

**픽업 장소:** 인천공항 T1·T2 출발층(3F) 게이트 앞 도로변 (초기 데이터 5곳)

**3앱 구성 (Turborepo 모노레포):**

| 앱 | 기술 | 대상 |
|---|---|---|
| `admin-web` | Next.js + TypeScript | 운영 관리자 |
| `user-app` | React Native (Expo) | 공항 이용 고객 |
| `driver-app` | React Native + **Android(Kotlin)** | 발렛 기사 |

**DB 스키마 (Supabase Postgres):**
- `users` (고객) · `vehicles` (등록 차량) · `drivers` (기사, 상태: available/busy/off_duty)
- `parking_zones` (구역: general/premium/partner) · `parking_slots` (주차면, 상태: available/occupied/reserved/maintenance)
- `reservations` (예약, 고객↔차량↔기사 연결) · `pickup_locations` (ICN T1/T2 접선 장소)

**기술 스택:** Turborepo, Next.js, React Native (Expo), Kotlin (Android), Supabase (Postgres + Auth + Storage), TypeScript

**최종 업데이트:** 2026-03-03
**상태:** 개발 중
