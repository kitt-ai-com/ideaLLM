### 핵심 주장: 우리는 이미 동일한 파이프라인을 운영 중입니다

대부분의 참가팀이 "이런 시스템을 만들겠다"는 계획을 제출합니다.
킷에이아이는 다릅니다. **이번 과제와 구조적으로 동일한 AI 문서 자동생성 파이프라인이 지금 이 순간에도 운영 중**입니다.

#### 현재 운영 중인 파이프라인 — newsletter-agent

```
[소스 수집]          RSS 피드 4개 (Anthropic·OpenAI·Google AI·TechCrunch)
      ↓                           ↕ 동일 단계
[주제 선정]          Claude: 수집 항목 → 테마·각도·소스 선정
      ↓
[구조 설계]          Claude: 테마 → 섹션 개요 JSON 생성
      ↓
[초안 생성]          Claude: 개요 + 소스 → 한국어 에세이 1,500~2,500자
      ↓
[메타데이터 추출]    Claude: 본문 → 제목·슬러그·발췌·카테고리 자동 생성
      ↓
[사람 승인 게이트]   터미널 미리보기 → 승인/재시도/취소 (자동 게시 경로 없음)
      ↓
[DB 게시]            Supabase INSERT → 웹사이트 즉시 반영
```

**이것이 PolicyDoc Agent와 다른 점은 소스와 출력 형식뿐입니다.**

| 구분 | newsletter-agent (현재 운영) | PolicyDoc Agent (이번 과제) |
|---|---|---|
| 입력 | RSS 피드 URL | 중진공 공개 데이터 + 사용자 키워드 |
| LLM | Gemini API (현재) / Claude API 전환 가능 | Claude API or Gemini (provider 선택) |
| LLM 추상화 | `LlmClient` 인터페이스 — 공급사 교체 1파일 | 동일 인터페이스 재사용 |
| 검증 | Zod 스키마 (단계별) | Zod 스키마 (동일 구조) |
| 승인 | 터미널 사람 확인 | 웹 미리보기 사람 확인 |
| 출력 | Supabase JSON → 웹 | HWP/PDF → 다운로드 |
| 코드 상태 | **GitHub 운영 중 (2026-05-18 최신)** | 10주 PoC 구현 |

> 새로 개발이 아닙니다. **검증된 파이프라인의 도메인 이전**입니다.

## 2. 경쟁 우위 — 왜 킷에이아이인가

### 다른 참가팀과의 결정적 차이

| 구분 | 일반 참가팀 | 킷에이아이 |
|---|---|---|
| 증거 | "이런 시스템을 만들겠습니다" | **동일 구조 파이프라인이 GitHub에서 운영 중** |
| LLM 경험 | API 연결 계획 | Claude·Gemini 실운영, 토큰 비용 최적화 완료 |
| 한국어 문서 생성 | 계획 단계 | 1,500~2,500자 에세이 자동 생성 **실운영 중** |
| 보안 | 계획서 기재 | 병원 환자 데이터 Zero Data Retention **실운영 실적** |
| 개발 속도 | 10주 전부 신규 개발 | 기존 모듈 재사용 → **선두 2~3주 확보** |
| HWP 대응 | 미언급 또는 계획 | python-hwp 전문가 자문 확보 예정 명시 |

### 킷에이아이가 유리한 3가지 구조적 이유

**① 파이프라인이 이미 있다**
newsletter-agent의 collect→draft→validate→review→publish 7단계가 PolicyDoc Agent와 1:1 대응됩니다. 타 팀이 파이프라인 설계부터 시작할 때 킷에이아이는 **도메인 변환만 하면 됩니다.**

**② B2B 납품 전주기를 이미 밟았다**
물류·병원·학원 3건에서 요구사항 수집 → 구축 → 인도 → 운영까지 완료했습니다. 10주 챌린지에서 발주처(중진공) 커뮤니케이션, 요구사항 변경 대응, UAT 운영이 가장 어렵습니다. 킷에이아이는 이 과정을 3번 이미 겪었습니다.

**③ 최종 평가가 "사용자 체험"이다**
심사 방식이 발표만이 아닌 **중진공 직원 실사용 평가**입니다. 서비스가 실제로 30초 안에 동작하고, 오류 없이 HWP가 다운로드되어야 합니다. 이 부분에서 "계획서만 있는 팀"과 "실제로 만들어 본 팀"의 차이가 극대화됩니다.

## 3. 문제 정의

### 2-1. 중소기업의 3가지 구조적 장벽

**장벽 1. 시간 — 대표가 직접 써야 하는 3시간**
- 사업계획서 1건 완성 평균 3시간 이상
- 전담 인력 없는 소기업은 대표 직접 작성 → 경영 공백
- 연 약 20,000건 기준 → 산업 전체 **60,000시간+ 낭비**

**장벽 2. 비용 — 외부 대행의 역설**
- 외부 컨설팅 의존 시 건당 수십~수백만 원
- 정책자금이 더 필요한 영세 기업일수록 접근 비용이 높은 역설

**장벽 3. 리스크 — 대행으로 인한 사실 왜곡**
- 제3자 대행 과정에서 실제와 다른 내용 기재 가능
- 사후 감사·자금 회수 위험

### 2-2. 중진공 관점: 신청서 품질 저하 → 심사 부담 가중

- 불완전 신청서 → 보완 요청 반복 → 담당자 업무 과부하
- 동일 작성법 문의 반복 → 핵심 심사 집중도 저하
- 대행 의심 신청서 별도 검증 비용 증가

**킷에이아이 솔루션은 기업과 중진공 양쪽의 부담을 동시에 해결합니다.**

## 4. 솔루션 개요

### kitt PolicyDoc Agent
#### AI 기반 정책자금 융자신청서 자동생성 서비스

**한 줄 정의:**
> 중소기업 대표가 키워드 5~10개를 입력하면, **30초 안에** 중진공 공식 서식에 맞는 사업계획서 초안을 자동 생성하는 반응형 웹 서비스

**서비스 3원칙:**

| 원칙 | 구현 | 보장 |
|---|---|---|
| **Fact-Grounded 생성** | 중진공 공식 공고문·서식·예시만 RAG 소스 | 외부 정보·허위 내용 생성 불가 |
| **서식 완전 보존** | HWP/PDF 원본 레이아웃·폰트·여백 100% 유지 | 별도 편집 불필요 |
| **Zero Data Retention** | 다운로드 완료 즉시 Secure Delete | 개인정보 잔류 없음 |

### 과제 제공 데이터 (중진공 → 킷에이아이)

10주 개발 실증 기간에 중진공이 직접 제공하는 공식 데이터:

| 데이터 | 형태 | RAG 활용 방식 |
|---|---|---|
| 정책자금 공고문 | PDF | 자금 유형별 요건·기준 벡터화 |
| 융자 신청서 서식 | HWP | 항목 구조·레이아웃 템플릿 추출 |
| 사업계획서 예시 | HWP (제조/비제조 샘플) | Few-Shot 예시 문서로 활용 |
| 가이드라인 매뉴얼 | PDF | 작성 요령·주의사항 컨텍스트 주입 |
| 기업 기본정보 실시간 DB | API (연동 동의 기반) | SSO 로그인 시 자동 매핑 |

> 공식 데이터만 RAG 소스로 사용하므로 할루시네이션 1층 방어가 구조적으로 보장됩니다.

## 5. 기술 구조

### 4-1. 전체 서비스 흐름

```
┌──────────────────────────────────────────────────────┐
│  [사용자 입력]                                          │
│  업종 선택 + 자금 유형 + 핵심 키워드 5~10개              │
└───────────────────────┬──────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────┐
│  [서식 분류 엔진]                                       │
│  업종 × 자금유형 → 중진공 공식 서식 자동 매칭              │
└───────────────────────┬──────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────┐
│  [RAG 파이프라인]                                       │
│  중진공 공고문 + 서식 + 예시 → 벡터 검색 + 컨텍스트 주입  │
│  ※ newsletter-agent collect 단계와 동일 구조            │
└───────────────────────┬──────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────┐
│  [LLM 초안 생성]                                        │
│  Claude API + 프롬프트 엔지니어링 + Few-Shot + CoT       │
│  항목별 스트리밍 출력 (첫 응답 5초 이내)                   │
│  Zod 스키마 검증 + 실패 시 3회 자동 재시도                 │
│  생성 문장마다 참조 서식 번호 자동 태깅                     │
│  ※ newsletter-agent draft·metadata 단계와 동일 구조     │
└───────────────────────┬──────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────┐
│  [사람 검토·수정]                                       │
│  웹에디터 항목별 편집 + 재생성(최대 5회/항목)              │
│  임시 저장 + 미리보기 + 진행률 표시                        │
│  ※ newsletter-agent review 단계와 동일 원칙             │
└───────────────────────┬──────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────────┐
│  [서식 출력 + 보안 삭제]                                 │
│  HWP / PDF 원본 레이아웃 보존 변환 → 다운로드             │
│  다운로드 완료 즉시 임시 데이터 Secure Delete             │
└──────────────────────────────────────────────────────┘
```

### 4-2. newsletter-agent 파이프라인과의 구조 대응

킷에이아이가 이미 운영 중인 newsletter-agent의 7단계 파이프라인이 이번 과제의 기술 청사진입니다.

```
newsletter-agent (운영 중)          PolicyDoc Agent (이번 과제)
─────────────────────────────────────────────────────────────
collect: RSS 피드 수집           →  collect: 중진공 공개 데이터 파싱
select_topic: 주제 선정 (Claude)  →  form_match: 서식·자금유형 분류
outline: 섹션 구조 설계 (Claude)  →  outline: 신청서 항목 구조화
draft: 본문 생성 (Claude, Zod)   →  draft: 항목별 초안 생성 (Claude, Zod)
metadata: 제목·슬러그 추출        →  metadata: 기업정보·서식코드 추출
review: 사람 승인 게이트           →  review: 웹 미리보기 + 수정
publish: Supabase INSERT         →  export: HWP/PDF 다운로드
```

**달라지는 것:** 입력 소스(RSS→공공데이터), 출력 형식(JSON→HWP)
**달라지지 않는 것:** Claude API, Zod 검증, 단계별 영속화, 사람 승인 게이트, 함수형 파이프라인

### 4-3. 할루시네이션 3중 방어 전략

> 심사위원이 가장 우려하는 문제를 정면으로 해결합니다. 회피하지 않습니다.

| 방어층 | 구현 | 효과 |
|---|---|---|
| **1층: RAG 소스 제한** | 중진공 공식 문서만 인덱싱, 외부 URL 차단 | 공식 범위 외 생성 물리적 불가 |
| **2층: 출처 태깅** | 생성 문장마다 참조 서식 번호·조항 병기 | 심사자가 출처 즉시 검증 가능 |
| **3층: Fallback** | 확신도 임계값 미달 항목 → "직접 입력" 안내 | 불확실한 내용 강행 없음 |

newsletter-agent에서 동일한 원칙을 적용 중: Zod 스키마 검증 실패 시 LLM에 위반 메시지 포함해 3회 재시도, 그래도 실패 시 파이프라인 중단.

### 4-4. 기술 스택

| 구성 요소 | 기술 선택 | 근거 |
|---|---|---|
| LLM | Claude API (Anthropic SDK) | 한국어 장문 생성 최상위, **실운영 경험 보유** |
| 스키마 검증 | Zod | newsletter-agent에서 동일 구조 운영 중 |
| 클라우드 | CSAP 인증 IaaS — **KT Cloud** | 공공 보안 요건 충족, 확정 |
| RAG | 벡터DB + 임베딩 모델 | 저지연 문맥 검색 |
| HWP 출력 | python-hwp + 레이아웃 엔진 | 원본 서식 100% 보존 |
| 인증 | 중진공 고객포털 SSO | 기업 기본정보 자동 연동 |

### 5-3. 6단계 사용자 시나리오

```
Step 1. SSO 로그인
  → 중진공 고객포털 재인증 없이 접근
  → 업종·기업규모·매출액·대표자 정보 자동 연동

Step 2. 자금 유형 + 키워드 입력
  → 자금 유형 선택 (일반경영안정자금 / 신시장진출자금 등)
  → 업종별 추천 키워드 + 핵심 키워드 5~10개 자유 입력

Step 3. AI 초안 생성 (30초 이내)
  → 서식 자동 매칭 → RAG + 프롬프트 → 항목별 스트리밍 출력
  → 각 문장에 참조 서식 번호 자동 태깅
  → Zod 검증 실패 시 자동 재시도 (최대 3회)

Step 4. 초안 수정·보완
  → 항목별 직접 편집 + AI 재생성 (최대 5회/항목)
  → 임시 저장 (세션 종료 후 24시간)

Step 5. 미리보기 + 다운로드
  → HWP 원본 레이아웃 미리보기
  → 다운로드 → 서버 임시 데이터 즉시 Secure Delete

Step 6. 피드백 (선택)
  → 별점·의견 → 품질 개선 반영
```
